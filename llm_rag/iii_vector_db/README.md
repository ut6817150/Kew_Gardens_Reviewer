# iii_vector_db

## Purpose

`llm_rag/iii_vector_db/` contains the reference retrieval layer for the RAG
system.

This stage reads preprocessed records from
`llm_rag/ii_preprocessed_documents/`, builds queryable retrieval assets, and
serves reference-side evidence at inference time. It is responsible for both:

- deterministic threshold lookup for stable Red List numeric facts
- hybrid dense plus sparse retrieval for broader reference evidence

## Main Files

| File | Purpose |
|---|---|
| `build_reference_db.py` | Build script that converts preprocessed retrieval blocks into sparse JSONL assets, parent table contexts, and an optional persistent Chroma dense index. |
| `retrieval_engine.py` | Query-time retrieval engine. Handles deterministic routing, dense retrieval, sparse BM25-style retrieval, score fusion, reranking, parent-context attachment, supporting-information query expansion, and answer scaffolds. |
| `threshold_lookup.py` | Deterministic threshold lookup for stable criteria facts such as AOO, EOO, locations, mature individuals, and extinction probability. |
| `embedding_loader.py` | Embedding-model loader with cached-snapshot resolution and sparse-only fallback support for query time. |
| `reference_retrieval_smoke_test.py` | Command-line smoke test for manually inspecting reference-side retrieval output. |
| `reference_corpus.jsonl` | Sparse-retrieval corpus generated by the build script. |
| `parent_contexts.jsonl` | Unchunked parent table contexts generated by the build script. |
| `thresholds.json` | Manually maintained deterministic threshold data. |
| `build_summary.json` | Generated summary of the latest reference-index build. |
| `chroma_db/` | Persistent Chroma dense-index assets. |

## Building The Reference Assets

Run from the repo root:

```bash
python llm_rag/iii_vector_db/build_reference_db.py --reset
```

Useful options:

```bash
python llm_rag/iii_vector_db/build_reference_db.py --skip-chroma
python llm_rag/iii_vector_db/build_reference_db.py --chunk-size 800 --chunk-overlap 100
```

Use `--reset` when rebuilding the dense index after preprocessing, chunking, or
embedding changes. Use `--skip-chroma` when only the JSONL assets are needed,
for example in a lightweight test or inspection run.

## Retrieval Routes

### Deterministic threshold lookup

`threshold_lookup.py` handles questions that map cleanly to stable numeric
threshold facts.

Examples include:

- EOO thresholds under Criterion B1
- AOO thresholds under Criterion B2
- number-of-locations thresholds under Criterion B
- mature-individual thresholds under Criteria C and D
- extinction-probability thresholds under Criterion E

When `answer_query(...)` is called directly and the threshold route returns an
answer, hybrid retrieval can be skipped. During full RAG inference,
`answer_rag_question(...)` calls the reference engine with threshold
short-circuiting disabled so deterministic facts and hybrid reference evidence
can both be included in the final prompt.

### Hybrid dense plus sparse retrieval

For broader questions, `retrieval_engine.py` combines:

- dense Chroma retrieval over embedded chunks
- sparse BM25-style retrieval over `reference_corpus.jsonl`
- reciprocal-rank fusion
- metadata boosts for table rows and high-value sources
- query-intent boosts for criteria, mapping, threats, conservation, habitats,
  and supporting information
- parent table-context attachment for row-level hits

This hybrid design matters because IUCN questions often mix semantic language
with exact symbols such as `AOO`, `EOO`, `D2`, `Criterion B`, and table names.

## Supporting-Information Query Expansion

Supporting-information questions often need more than one evidence type. For
example, a question may need both:

- requirements that apply to all assessments
- additional requirements under specific criteria or conditions

When the retriever detects this query type, it expands the user question into
internal subqueries and tries to force coverage for key supporting-information
tables. This is why debug output may show several internal subqueries for one
visible prompt.

## Manual Smoke Test

Run a quick retrieval check from the repo root:

```bash
python llm_rag/iii_vector_db/reference_retrieval_smoke_test.py "What supporting information is required for a threatened species assessment?"
```

The smoke test prints:

- selected route
- deterministic threshold answer, if present
- internal subqueries
- answer scaffold
- retrieved candidates
- parent table context when available

This is a quick diagnostic tool, not a formal benchmark.
