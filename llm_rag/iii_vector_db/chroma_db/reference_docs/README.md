# chroma_db/reference_docs

## Purpose

This folder stores the persistent Chroma collection named:

```text
iucn_reference_docs
```

It is the dense-vector index for the reference documents. The retrieval engine
uses it when dense embeddings are available.

## Current Contents

Typical contents include:

- `chroma.sqlite3`
  Chroma metadata and collection database.
- UUID-named subfolders
  HNSW/index files for stored embeddings.
- binary index files such as `data_level0.bin`, `header.bin`, `length.bin`,
  `link_lists.bin`, and `index_metadata.pickle`
- `README.md`
  This document.

The UUID folder names are generated by Chroma and should not be interpreted as
semantic document identifiers.

## How It Is Generated

The folder is generated by:

```bash
python llm_rag/iii_vector_db/build_reference_db.py --reset
```

The build script:

1. reads `llm_rag/ii_preprocessed_documents/*/retrieval_blocks.jsonl`
2. splits records into indexable chunks
3. builds embeddings with `BAAI/bge-m3`
4. writes the persistent Chroma collection here
5. writes `llm_rag/iii_vector_db/build_summary.json`

## Runtime Behavior

At query time, `llm_rag/iii_vector_db/retrieval_engine.py` opens this directory
through Chroma. If the dense index or embedding model is unavailable, the
retriever can fall back to sparse retrieval from `reference_corpus.jsonl`.

That fallback keeps the app usable, but dense retrieval quality depends on this
folder being present and in sync with the current JSONL assets.

## Commit And Cleanup Notes

This folder is generated, but it is not the same kind of disposable artifact as
`__pycache__` or `.pytest_cache`. It may be needed in deployments that should
start without rebuilding the dense index.

Delete it only when:

- rebuilding with `build_reference_db.py --reset`
- intentionally removing generated vector assets from the repository
- replacing it with a freshly built Chroma collection
